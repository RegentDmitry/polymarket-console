# Weather Markets — Стратегия и План

## Идея

Polymarket предлагает ежедневные рынки на максимальную температуру в 16 городах мира.
Каждый рынок разбит на 7-9 температурных бакетов (например "44-45°F", "46-47°F").
Прогнозы погоды от NWP-моделей (GFS, ECMWF, ICON, JMA) на 1-3 дня вперёд дают
точность значительно выше чем PM-трейдеры. Типичный edge 10-50%.

## Бэктест (351 закрытый рынок, 2457 бакетов)

Проведён на всех закрытых температурных рынках PM (Jan-Mar 2026).

**Ground truth: IEM METAR/ASOS** (те же сенсоры, что использует Weather Underground).
До марта 2026 использовался ERA5 (Open-Meteo archive), который давал Brier=0.1183.
Переход на IEM дал **Brier=0.0750 (улучшение 37%)**.

## Модель

```
fair_price(bucket) = P(lower ≤ Normal(forecast, σ) < upper)
```

- **forecast** = mean(daily_max по 4 NWP-моделям) из Open-Meteo ensemble
- **σ** = max(σ_ensemble, σ_floor) где:
  - σ_ensemble = std(daily_max по моделям) — разброс между GFS/ECMWF/ICON/JMA
  - σ_floor = 2.5°F / 1.39°C (IEM-calibrated: sqrt(σ_iem² + σ_forecast²))
- **edge** = fair_price - pm_yes_price

## Стратегия торговли

### Buy → Hold to Resolution

**Sell at fair НЕ подходит для погоды** (в отличие от землетрясений):

| Фактор | Землетрясения | Погода |
|--------|--------------|--------|
| Горизонт | месяцы | **1-3 дня** |
| Fair price | стабильна | **меняется каждые 6ч** (новый прогноз) |
| Fill rate | высокий (время есть) | **низкий** (кто купит по fair если прогноз открыт?) |
| Turnover | медленный (нужен sell) | **быстрый** (resolution через день) |

**Правильная стратегия:**
1. **Buy** когда edge > min_threshold
2. **Hold** до resolution (следующий день)
3. **Sell** только если прогноз резко изменился и edge пропал

Капитал оборачивается ежедневно — нет нужды в раннем sell.

### Исключение: sell при смене прогноза

Если после покупки выходит новый forecast run (каждые 6ч) и edge стал отрицательным —
продать в убыток лучше чем держать до resolution.

## Сканер

### Скорость

- **Полный скан: ~100 сек** (16 городов × Open-Meteo ensemble + Gamma API)
- Bottleneck: Open-Meteo API (~5 сек/город, 4 модели)
- С кэшированием forecast: **~5-10 сек** (только PM цены обновляются)

### Оптимальная частота обновления

| Компонент | Частота | Почему |
|-----------|---------|--------|
| **Forecast** | 4 раза/день (00, 06, 12, 18 UTC) | GFS/ECMWF обновляются каждые 6-12ч |
| **PM цены** | каждые 5-10 минут | Дёшево (1 HTTP запрос), цены двигаются |
| **Полный скан** | каждый час | Баланс между точностью и нагрузкой |

### Лучшее время для покупки

- **Утро UTC (06-10)**: новые прогнозы вышли ночью, PM трейдеры ещё не обновили цены
- **Вечер перед resolution**: σ эффективно ниже (меньше времени до resolution), edge точнее
- Избегать: первые часы после открытия рынка (ликвидность тонкая)

## Риски

1. **Forecast error** — реальный σ выше чем σ_iem=1.0°F
   - Митигация: σ_floor=2.5°F включает forecast error + ensemble spread
2. **IEM ≠ WU точно** — bias ±1°F для US, outliers в NYC до 22°F
   - Митигация: IEM уже значительно лучше ERA5 (66.7% vs 28.5% bucket hit)
3. **Ликвидность** — $300-3k на бакет, реально $100-400
   - Митигация: распределять по городам и датам
4. **Прогноз меняется** — купил утром, вечером прогноз другой
   - Митигация: мониторить forecast updates, sell если edge пропал
5. **Tail events** — шторм, heatwave
   - Митигация: ensemble spread автоматически увеличивает σ
6. **PM меняет единицы** — London переключился с °C на °F
   - Митигация: unit определяется из question text, не из конфига
